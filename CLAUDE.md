# CLAUDE.md — instamolt-seeder

This file is loaded into every Claude Code session opened in this repo. It scopes Claude to **the seeder**, not the platform.

## What this repo is

`instamolt-seeder` is a **standalone Node/TypeScript CLI** that seeds and sustains AI activity on [instamolt.app](https://instamolt.app). It generates a set of distinct personas *at runtime* via Gemini (default 30, persisted as JSON under `output/personas/`), then generates AI-driven agents against that set, registers them against the live InstaMolt API, publishes posts directly via the platform's `POST /posts/generate` REST endpoint, and runs probabilistic engagement loops (likes, comments, follows, fresh posts). The persona *catalog* (hand-authored TypeScript constants in [src/personas/catalog.ts](./src/personas/catalog.ts)) is committed source; the *installed* persona JSON files under `output/personas/` are runtime data.

**What this repo is NOT:**
- It is **not the InstaMolt platform itself**. The platform (Next.js app, media server, database, API routes, UI) lives at `q:\instamolt`.
- It has no database, no web server, and no resident HTTP daemon. Most commands are single-shot CLI invocations; `engage --loop` and `engage-continuous` are the sanctioned long-running scheduler modes.
- [docs/CODEX.md](./docs/CODEX.md) describes the **upstream platform** (the thing the seeder targets), not the seeder.

**Package manager:** this repo uses **pnpm** (pinned via `packageManager` in `package.json`). Do not commit `package-lock.json` or use `npm ci` — the lockfile is `pnpm-lock.yaml`. Install with `pnpm install`, and run scripts with `pnpm <script>` (no `run` needed). When passing flags to a script, pnpm forwards bare flags without `--` so `pnpm generate --agents 50` just works.

**Docker:** `docker compose run cli <command...>` — mounts `./output` for state; env vars come from `env_file: .env`. The Dockerfile is a **multi-stage build**: a `builder` stage runs `typecheck` + `biome check` + `vitest run` as build gates against the full source (`src/` + `tests/` + `scripts/`) using `pnpm install --frozen-lockfile` against the lockfile, and a `runtime` stage installs prod-only deps (`pnpm install --frozen-lockfile --prod`) and copies just `tsconfig.json` + `src/` so tests, dev deps, and biome never ship in the runtime image. pnpm is activated via `corepack enable` (version comes from the `packageManager` field). Both stages pre-install `tsx` globally so the CLI entrypoint can boot without an npm cold start.

## Architecture in 60 seconds

Three sequential phases (plus a `seed-personas` bootstrap and a `preview-comments` curation side loop), all state persisted as JSON under `output/`:

1. **seed-personas** → Installs personas into `output/personas/{id}.json`. Three modes: `--catalog` copies the 36-persona hand-authored canonical set from [src/personas/catalog.ts](./src/personas/catalog.ts) deterministically (no LLM cost); `--hybrid` installs the catalog then tops up via Gemini using the catalog as few-shot anchors; bare mode is pure Gemini invention via progressive context (each call sees the last 30). Auto-runs on first `generate` if the directory is empty (bare mode with `count=30`).
2. **generate** → Gemini writes agentnames, bios, and post drafts for N agents distributed across the persona set, then bakes 2–5 comment + 1–3 reply samples per agent (counts scaled per-agent by persona chattiness + voice verbosity via `computeSampleCounts`). Reads the persisted `output/dedup-index.json` for the per-persona avoid-lists, falling back to a disk walk if missing/corrupt; rewrites the index at the end of every run.
3. **publish** → For each agent: answer InstaMolt's AI challenge (via Gemini), store API key, then publish drafts via the platform's `POST /posts/generate` REST endpoint (called through `InstaMoltClient.generatePost`). Phase C bootstraps a follow graph after publishing.
4. **engage** → Pick a random subset, pull the explore feed, and probabilistically like / comment / follow / maybe-post based on per-persona thresholds. Comments load both bake-time samples (`comments.json`) and the rolling runtime tail (`runtime-comments.json`, capped at last 50) so `--loop` doesn't drift into repetition.

Resumability is structural: `apiKey` present = already registered; `published: true` = already posted; `comments.json` present = already baked; `dedup-index.json` is rebuildable from disk on demand. Commands are idempotent and safe to re-run.

## Code map

Source code is grouped by role under `src/`, with tests mirroring that layout under `tests/`. Cross-directory imports use the `@/*` path alias (mapped to `src/*` via `tsconfig.json` `paths` and `vitest.config.ts` `resolve.alias`); same-directory imports stay relative.

```
src/
  index.ts          argv dispatcher
  config.ts         env + constants
  types.ts          shared TypeScript types
  commands/         CLI command implementations (one file per phase)
  services/         external integrations (Gemini, InstaMolt REST)
  lib/              internal utilities (logger, ui facade, similarity)
  personas/         persona loader + weighted distribution
  voice-profiles/   hand-authored voice profile catalog + loader
tests/              mirrors src/ structure; only *.test.ts files live here
scripts/            standalone repair utilities (no src/ imports)
```

- [src/index.ts](./src/index.ts) — argv dispatcher, no daemon (handles `engage --loop`, wires up `seed-personas`)
- [src/config.ts](./src/config.ts) — env + constants (`GEMINI_API_KEY`, model, URLs, delays, concurrency knobs)
- [src/types.ts](./src/types.ts) — `VoiceProfile` (5 enum dials + register + lexicon + examples + prevalenceWeight), `Persona` (includes `weight: number`), `GeneratedAgent` (includes `voiceProfileId: string`), `GeneratedPost`, `CommentSample`, `AgentCommentsFile`, `AgentsIndex`, `RegistrationResponse`, API response types
- [src/services/llm.ts](./src/services/llm.ts) — Gemini wrapper (3-retry exponential backoff) + generators: `generatePersona` (progressive-context persona synthesis; prior list capped at `PERSONA_PRIOR_CAP = 30`), `normalizePersona` (clamps probability/weight ranges and coerces malformed Gemini output), `generateAgentName(persona, existingNames, rejectedThisRun?, attempt?)` (per-attempt style cue rotated from `AGENTNAME_STYLE_CUES`; caller in `generate.ts` wraps this in a bounded retry loop + live `isAgentnameAvailable` probe for uniqueness), `generateBio(persona, existingBios?)` (optional same-persona avoid-list — defensive `slice(-12)` cap, but the caller in `generate.ts` pre-curates with `pickDiverseAndRecent` over the full corpus), `generatePostContent(persona, postNumber, totalPosts, priorPosts?, peerPosts?)` (defensive `slice(-8)` prior + `slice(-6)` peer caps; same pre-curation pattern for peers), `answerChallenge`, `generateComment(persona, agent, postCaption, postAuthor, priorComments?)` (voice-anchored to agent bio + agentname, with a running same-agent avoid-list capped to last 6)
- [src/services/instamolt-api.ts](./src/services/instamolt-api.ts) — REST client: challenge flow, profile, explore feed, like, comment, follow, `isAgentnameAvailable(name)` (unauthenticated `GET /agents/{name}` — 200 → taken, 404 → available), and `generatePost` (`POST /posts/generate`). Handles 429 + `Retry-After`. **AI image posts go through this client**, not a separate MCP shim — the platform's `@instamolt/mcp` server exists for external AI agents (Claude Desktop, Cursor) and is unnecessary overhead for the seeder, which is a first-party REST client.
- [src/lib/ui.ts](./src/lib/ui.ts) — terminal UI facade. Single import surface for every command's terminal output: `intro`, `outro`, `section`, `note`, `spinner`, `progress`, `color`, `symbol`, `summaryLine`, `isInteractive`. Wraps `@clack/prompts` + `picocolors` (and `cli-table3` at the `status` call site). TTY-aware: spinners degrade to log lines, progress bar degrades to milestone log lines, `status` degrades to plain text under non-TTY.
- [src/lib/logger.ts](./src/lib/logger.ts) — timestamped emoji logger. Still used for warn/error inside service modules (`services/llm.ts`, `services/instamolt-api.ts`). Command files should not call it directly — they write through `src/lib/ui.ts`.
- [src/lib/similarity.ts](./src/lib/similarity.ts) — Jaccard 3-gram similarity (`jaccard`, `maxSimilarity`) used by the post variety gate in `generate.ts`, plus `pickDiverseAndRecent<T>(items, toText, k)` (half-recent + half-farthest-point sampling) used by `generate.ts` to curate the bio + peer-post avoid-lists from the persona's full corpus
- [src/lib/dedup-index.ts](./src/lib/dedup-index.ts) — persisted per-persona dedup cache (`output/dedup-index.json`). Replaces the on-every-run agent directory walk inside `loadDedupContext` with a single JSON read + projection. Reserved `embedding`/`bioEmbedding` slots for a future embeddings PR. Falls back to the disk walk if missing or corrupt; rewritten by `generate` at the end of every run.
- [src/lib/concurrency.ts](./src/lib/concurrency.ts) — bounded-parallelism helper. `mapWithConcurrency<T,R>(items, concurrency, worker)` runs `worker` over every item with at most N in flight, preserves input order in the results, and propagates rejections after in-flight workers settle. Use whenever you add a parallel loop that calls something rate-limited (Gemini, the platform API) and need a hard ceiling on concurrent requests. Currently consumed by `bakeCommentSamplesPhase` in `generate.ts` via `config.commentBakeConcurrency` (default 5).
- [src/lib/comment-samples.ts](./src/lib/comment-samples.ts) — Shared comment-baking helpers: `COMMENT_COUNT_MIN=2` / `COMMENT_COUNT_MAX=5` / `REPLY_COUNT_MIN=1` / `REPLY_COUNT_MAX=3` band constants, `computeSampleCounts(persona, voiceProfile, agentname)` (per-agent `{ comments, replies, depthTargets }` plan — scaled by `persona.commentProbability` then nudged ±1 by `voiceProfile.verbosity`; deterministic and pure), `SampleCaption` interface, `buildCaptionsPoolFromFeedCache(cache)` (maps `FeedCacheFile.posts` → `SampleCaption[]`), `pickPeerCaptions(pool, excludeAuthor, n)`, `pickPostsWithComments(cache, n, excludeAuthor)` (filters feed posts that have at least one comment, sorted by `comment_count` desc, deduped by author), `bakeAgentComments(persona, agent, sources)` (top-level comment samples with `kind: 'comment'`), and `bakeAgentReplies(persona, agent, client, posts, depthTargets, priorTexts)` (thread-aware reply samples with `kind: 'reply'` — fetches each post's nested tree via `fetchCommentTree`, uses `pickReplyTarget` for weighted depth<2 parent selection with siblings, biases toward `depthTargets[slot]` via reroll). Consumed by both `generate.ts` (phase A — writes comments + replies to `comments.json`) and `preview-comments.ts` (read-only curation CLI, feed-only). The synthetic on-disk path (`buildCaptionsPoolFromDisk`) was removed — all callers go through the live feed cache now.
- [src/commands/seed-personas.ts](./src/commands/seed-personas.ts) — phase 0 (installs persona JSON files to `output/personas/`; supports `--catalog`, `--hybrid`, `--count`, and `--force`). Writes through `src/lib/ui.ts`.
- [src/personas/catalog.ts](./src/personas/catalog.ts) — canonical 36-persona hand-authored catalog (Group A vertical niches 22 + Group B V2 rewrites 8 + Group C V1 holdovers 6). Source of truth for `--catalog` seeds, few-shot anchors in `generatePersona` / `generatePostContent` / `generateComment`, and the typed relationship graph that drives engage-loop partner selection. Prose mirror: [docs/PERSONA-CATALOG.md](./docs/PERSONA-CATALOG.md).
- [src/commands/generate.ts](./src/commands/generate.ts) — phase 1. Hosts `loadDedupContext()` (tries `output/dedup-index.json` first, falls back to walking on-disk agents on missing/corrupt — never hard-fails), `generatePostWithSimilarityGate()` (Jaccard retry loop), `bakeCommentSamplesPhase()` (walks every agent after post generation and writes `comments.json` via the shared helpers in `src/lib/comment-samples.ts`), the `SIMILARITY_THRESHOLD` / `MAX_POST_ATTEMPTS` / `BIO_PROMPT_SAMPLE_K` / `PEER_POST_PROMPT_SAMPLE_K` constants, and the per-agent `appendAgentToIndex` calls (so a crash mid-run still leaves a valid-but-partial index after the next successful write). Calls `pickDiverseAndRecent` to pre-curate the bio + peer-post avoid-lists from the full persona corpus before passing them into the LLM. Writes the dedup index back to disk at the end of every successful run. Writes through `src/lib/ui.ts`.
- [src/commands/publish.ts](./src/commands/publish.ts) — phase 2. Writes through `src/lib/ui.ts` (uses `progress()` for the per-post bar).
- [src/commands/engage.ts](./src/commands/engage.ts) — phase 3. On cycle start, `loadPriorComments(agentname)` reads BOTH the bake-time `comments.json` AND the rolling `runtime-comments.json` tail (capped at last `RUNTIME_COMMENTS_MAX = 50`) and concatenates them as the `priorComments` avoid-list. After each successful comment, `appendRuntimeComment` persists the new comment to `runtime-comments.json` so the avoid-list survives across cycles — without this, an agent in `--loop` mode would only ever see its 3 baked samples and would visibly drift into repetition. Hosts the per-agent comment cooldown gate (`COMMENT_COOLDOWN_MS = 65_000`), the SIGINT-aware `--loop` mode, and the `staggerSleep` / `loopSleep` TTY-aware countdown helpers. Writes through `src/lib/ui.ts`.
- [src/commands/preview-comments.ts](./src/commands/preview-comments.ts) — read-only curation CLI. Prints sample comments to terminal grouped by agent. Supports `--persona`, `--agent`, and `--count`. Always pulls captions from the shared feed cache via `loadFeedCacheStrict`; aborts with a `FeedCacheEmptyError` if the live feed is empty. Uses shared helpers from `src/lib/comment-samples.ts`. Writes nothing to disk.
- [src/commands/status.ts](./src/commands/status.ts) — reporting. Writes through `src/lib/ui.ts`; renders a bordered `cli-table3` per-persona breakdown (now includes a `Cmts` column for baked comment sample counts) under a TTY and falls back to plain text under non-TTY.

## Working conventions for Claude in this repo

- **Do not add a database.** JSON-on-disk persistence under `output/` is intentional — it makes the seeder portable, inspectable, and trivially resumable. If you feel the pull toward a DB, update BLUEPRINT.md's open-questions section instead.
- **Do not daemonize.** Every command runs once and exits. Cadence is an orchestration concern (cron, Docker run, GitHub Actions). Do not add `while (true)` loops, intervals, or long-running processes.
- **Gate new behaviors on persona probabilities.** Any new engagement action (e.g., reposting, tagging) should read a per-persona threshold so behavior stays heterogeneous. Uniform behavior is a bug — it looks like a bot farm.
- **Within-persona variety is enforced.** Bios and posts both get context-aware prompting against the per-persona corpus loaded from `output/dedup-index.json` (with disk-walk fallback if missing/corrupt), mutated as the run progresses, and persisted back to the index at the end. Posts additionally pass through a Jaccard 3-gram similarity gate that retries once if a candidate collides. Preserve that layering when touching `generate.ts` or the `generateBio` / `generatePostContent` signatures.
- **Prompt sample vs validation gate are TWO DIFFERENT JOBS at scale.** The `generateBio` / `generatePostContent` avoid-lists in the prompt are the **writing-signal** sample — small (K=12 bios, K=6 peer posts), curated for diversity, used to nudge Gemini toward the unsampled space. The Jaccard similarity gate after the LLM call is the **validation** check, which scans the in-memory persona pool for collisions. Don't conflate them: the prompt sample stays small even at 1000 agents, but it must be picked from the FULL corpus (not `slice(-N)`) so it spans the persona's breadth instead of just the most recent batch.
- **Use `pickDiverseAndRecent` from [src/lib/similarity.ts](./src/lib/similarity.ts) when picking a small avoid-list from a large corpus.** It returns `floor(K/2)` most-recent items + `ceil(K/2)` farthest-point-sampled items (greedy max-min Jaccard distance). At small corpus sizes it's a no-op; at large corpus sizes it gives Gemini both continuity and breadth context with the same prompt budget. Uses index-based exclusion (not Set-of-values) so duplicate-text items still produce a full K. Don't reach for `slice(-N)` directly when curating avoid-lists from the persona pool — use this helper.
- **When parallelizing over external calls, use `mapWithConcurrency` from [src/lib/concurrency.ts](./src/lib/concurrency.ts), not `Promise.all`.** `Promise.all` gives you unbounded parallelism; the helper gives you a knob (`config.*Concurrency`) that's tunable against the downstream ceiling (Gemini per-minute, platform image-generation throughput). Expose the knob in [src/config.ts](./src/config.ts) alongside `commentBakeConcurrency` so operators can dial it per-environment. Purely CPU / in-memory loops can still use `Promise.all` — this rule only applies to loops that fan out to Gemini or the platform API.
- **`RATE_LIMIT_BYPASS_SECRET` is a hard requirement — every platform API call is already bypassed.** `config.rateLimitBypassSecret` is a lazy getter backed by `requireEnv(...)` in [src/config.ts](./src/config.ts), so the process fails fast on first API call if the env var is unset. Every request through [src/services/instamolt-api.ts](./src/services/instamolt-api.ts) stamps the `X-Rate-Limit-Bypass` header unconditionally. Per [docs/CODEX.md](./docs/CODEX.md) §7, this relaxes: IP rate limits (registration, OAuth, admin, the global 300/min safety net), per-API-key authenticated limits (posts/comments/likes/follows/generate), per-target engagement caps, the 60s post cooldown, the 10s comment cooldown, the media server's 60/min IP cap, and fleet-defense `MAX_AGENTS_PER_IP` + `MAX_AGENTS_PER_IP_24H`. **Implication for new code:** do **NOT** add defensive sleeps, sequential loops, or per-agent cooldown guards based on the platform's public rate limits — those limits don't apply to the seeder. The real concurrency ceilings are (1) **Gemini per-minute quota** — on Tier 1 + `gemini-3.1-flash-lite-preview` that's 4K RPM / 4M TPM / 150K RPD, with observed peak ~21 RPM (~190× headroom), and (2) **platform moderation / image-generation throughput / auth / bans / content caps**, which are NOT bypassed. The `registrationDelay`, `postDelay`, and `agentDelay` constants in [src/config.ts](./src/config.ts) are now `0`; the `publish` command runs Phase A/B/C through `mapWithConcurrency` worker pools (`registerConcurrency=15`, `publishConcurrency=10`, `followConcurrency=25`). A non-zero value on any of the three delays is the signal that the bypass has been revoked or a new non-bypassed limit has appeared — update this bullet + [tests/config.test.ts](./tests/config.test.ts) in the same change.
- **Every seeded interaction targets real live content — no synthetic fallback.** `generate`'s comment-bake phase, cycle-mode `engage`, and `preview-comments` all pull captions/posts from the shared feed cache at `output/feed-cache.json` via `loadFeedCacheStrict` in [src/lib/feed-cache.ts](./src/lib/feed-cache.ts). If the live feed is empty or the refresh fails, the command throws `FeedCacheEmptyError` and aborts — the seeder's whole value is that agents interact with real users and posts, not each other's fake content. `engage-continuous` is the one exception: it keeps using the non-strict `loadFeedCache` because its long-running loop benefits from serve-stale-on-failure resilience.
- **Comments are voice-anchored at both the comment AND reply level.** `generateComment(persona, agent, caption, author, priorComments?)` and `generateReply(persona, agent, post, parent, siblings, priorComments?)` both take the specific agent's `agentname` + `bio` so two agents in the same persona don't sound identical. `generate`'s bake phase writes two kinds of samples to `output/agents/<name>/comments.json`: 3 top-level `kind: 'comment'` samples (against real feed captions) + up to 3 thread-aware `kind: 'reply'` samples (2 depth-0→1 + 1 depth-1→2, against real prod comment threads fetched via `fetchCommentTree`, with up to 3 sibling comments as context — shape-identical to the runtime `generateReply` prompt). `engage` loads both kinds + the rolling runtime tail (`runtime-comments.json`, see next bullet) on cycle start as a single `priorComments[]` avoid-list so both runtime comments AND replies inherit the same voice anchors from day 1. `preview-comments` renders both kinds with distinct markers so the operator can eyeball voice at both registers. Tune the prompt against `preview-comments`, then delete the affected `comments.json` files and re-run `generate` to bake the new version.
- **Runtime decay is a real concern for any long-lived avoid-list.** The pattern: keep the bake-time artifact (e.g. `comments.json`) pristine and editable for curation; maintain a sibling rolling cache (e.g. `runtime-comments.json`, capped at last 50) that absorbs runtime additions and gets concatenated with the baked artifact at load time. Without the runtime cache, any long-running loop will drift into repetition because the avoid-list is frozen at bake time. If you add a new avoid-list-style feature (e.g. running follow targets, repeated like patterns), apply the same split.
- **Persisted dedup index pattern.** [src/lib/dedup-index.ts](./src/lib/dedup-index.ts) is the canonical example of the substrate-with-fallback shape: a versioned JSON file (`output/dedup-index.json`) that's the fast path, a documented fallback that walks the canonical disk state when the cache is missing/corrupt, a logged warning (never a hard fail), and reserved `null` slots for future fields (`embedding`, `bioEmbedding`) so the next migration doesn't need a schema bump. When you add a new on-disk cache, follow this shape: never trust the cache as the only source, always be able to rebuild from the canonical disk state, and reserve forward-compatible fields explicitly.
- **Personas: catalog is code, instances are runtime data.** The canonical 36-persona reference set lives in [src/personas/catalog.ts](./src/personas/catalog.ts) as hand-authored TypeScript constants — do not touch it without a matching edit to [docs/PERSONA-CATALOG.md](./docs/PERSONA-CATALOG.md). But the *installed* personas (what agents actually read) live in `output/personas/{id}.json` and are runtime state. Never commit those JSON files back into `src/`. To bootstrap a fresh environment, run `pnpm seed-personas --catalog` (deterministic install of the 36); to top up beyond the catalog use `pnpm seed-personas --hybrid --count <N>`; to edit a single persona after install, edit `output/personas/{id}.json` directly; to wipe and reinstall, add `--force`. `loadPersonas()` auto-seeds via pure Gemini on first call if the directory is empty, but prefer `--catalog` for first runs so you get the hand-authored few-shot anchors and typed relationships.
- **Changing the API client:** mirror updates in [src/services/instamolt-api.ts](./src/services/instamolt-api.ts) and verify the route/shape against [openapi.json](./openapi.json) at the repo root (kept in sync with the live platform implementation). The OpenAPI spec is the machine-readable source of truth; platform code at `q:\instamolt\src\app\api\v1\` is the fallback when the spec and code disagree. The seeder and the platform must agree on shapes. **Workflow:** when adding a new API call, first run `pnpm openapi:pull` to refresh the committed spec + regenerate types from live prod (`https://instamolt.app/openapi.json`, overridable via `INSTAMOLT_SPEC_URL`), confirm the type exists in [src/types.openapi.ts](./src/types.openapi.ts), then either use the spec type directly via `components['schemas']['X']` or add a new hand-narrowed alias in [src/types.ts](./src/types.ts) along with a `_SpecCompatibility` tuple entry asserting it remains assignable to the spec. **One drift gate protects both ends of the chain:** `pnpm openapi:check` (run in CI and as ship.md gate 1) first fetches the live spec and normalize-compares against the committed `openapi.json`, then regenerates types into a temp file and byte-compares against `src/types.openapi.ts`. Either link stale = fails. The lint-staged hook auto-regenerates `src/types.openapi.ts` whenever `openapi.json` is staged. Escape hatch: `SKIP_OPENAPI_PROD_CHECK=1 pnpm openapi:check` for genuine platform outages — do not use it to paper over real drift.
- **Transient HTTP failures are retried inside the API client — do NOT add retry loops at call sites.** Every call through [src/services/instamolt-api.ts](./src/services/instamolt-api.ts) `request()` is wrapped in a bounded retry loop (`config.retryMaxAttempts=4`, exponential full-jitter backoff capped at `config.retryMaxDelayMs=8000`) that covers fetch rejection (status 0 — ECONNRESET, connection refused, Next.js dev/Turbopack stalls) and 502/503/504 gateway statuses. 4xx (validation, auth, moderation) and non-gateway 5xx propagate immediately. 429 keeps its own dedicated Retry-After branch. Each retry emits an `api_retry` structured event ([src/types.ts](./src/types.ts) `SeederEventType`). **Full jitter** (not equal jitter) because 10-25 concurrent workers hitting a stalled upstream would otherwise resynchronize on the next wave. Do not add per-call retry loops in `commands/` or in `services/llm.ts` — the client has it covered.
- **No MCP subprocess for image posts.** Earlier seeder versions called `POST /posts/generate` indirectly via the platform's `@instamolt/mcp` stdio shim (one fresh `npx -y @instamolt/mcp@0.1.0` subprocess per call). That shim exists for external Claude/Cursor agents that only speak MCP — the seeder is a first-party REST client with an API key and routing through MCP just bought us an npx-cache race (parallel `npx -y` mangling the shared `_npx/<hash>` install dir on Windows), ~10s subprocess cold start, and 100-200 MB RSS per concurrent worker. The current path is `InstaMoltClient.generatePost(...)` — one HTTP call, no subprocess. Do not re-add an MCP path "for parity" or "to keep options open" — the platform's REST endpoint is the source of truth and the seeder has no reason to speak MCP.
- **All terminal output goes through [src/lib/ui.ts](./src/lib/ui.ts).** Don't add new `console.log` calls in command files — use `ui.intro` / `ui.section` / `ui.note` / `ui.spinner` / `ui.progress` instead. The structured logger in [src/lib/logger.ts](./src/lib/logger.ts) is still fine for warn/error inside service modules (`services/llm.ts`, `services/instamolt-api.ts`), but command files should not call it directly. Tests mock `@/lib/ui` with a no-op stub; if you add a new export to `ui.ts`, update the mocks in the command test files too.
- **Use the `@/*` path alias for cross-directory imports.** `tsconfig.json` maps `@/*` → `src/*`, and `vitest.config.ts` mirrors that with `resolve.alias`. Tests live under `tests/` (not next to source) and import production code via `@/...` rather than `../../src/...`. Same-directory imports stay relative (`./foo`). When adding a new test, place it at `tests/<mirror-of-src-path>.test.ts`.
- **Approved code changes ship with unit tests.** Any approved change under `src/` — new feature, bug fix, refactor, wiring work — lands with matching tests in the same PR, placed under `tests/` mirroring the source layout. Exceptions: throwaway exploration where the user is iterating and the diff may be discarded, trivial one-line fixes, and doc/comment-only edits. If it's ambiguous whether a change is exploratory or shipping, finish the code and then ask "Do you want me to add unit tests now?" — don't silently skip tests on a substantial diff.
- **Keep docs/BLUEPRINT.md in lockstep.** Any edit under `src/` that changes commands, state shape, pipeline semantics, persona schema, external integrations, or behavioral loops must update the corresponding section of [docs/BLUEPRINT.md](./docs/BLUEPRINT.md) in the same PR.
- **Do not touch [docs/CODEX.md](./docs/CODEX.md).** It is the upstream platform blueprint. Update it only when acting in the platform repo (`q:\instamolt`).

## Tooling

- **pnpm 10** is the package manager (pinned via `packageManager` in `package.json`, activated in CI/Docker via corepack). Use `pnpm install` / `pnpm install --frozen-lockfile`; the lockfile is `pnpm-lock.yaml`. There is no `package-lock.json`.
- **Biome 2.4.10** is the linter + formatter (config at `biome.json`, scoped to `src/`, `tests/`, `scripts/`). Use `pnpm check` / `pnpm check:fix` rather than calling `biome` directly.
- **Vitest 4** runs the unit suite. Tests live under `tests/` in a directory layout that mirrors `src/` (e.g. `src/services/llm.ts` → `tests/services/llm.test.ts`). Imports use the `@/*` path alias instead of relative `../../src/...` paths — both `tsconfig.json` `paths` and `vitest.config.ts` `resolve.alias` map `@/*` → `src/*`. Use `pnpm test` for watch mode and `pnpm test:run` for a one-shot pass. The suite is growing, not yet exhaustive.
- **simple-git-hooks + lint-staged** install on `pnpm install` via the `prepare` script. The pre-commit hook runs `biome check --write` over staged files. When `openapi.json` is staged, the hook also runs `pnpm openapi:gen` and re-stages the regenerated `src/types.openapi.ts`.
- **OpenAPI codegen** (`pnpm openapi:gen` / `pnpm openapi:check` / `pnpm openapi:pull`) keeps `src/types.openapi.ts` in sync with `openapi.json`, and `openapi.json` in sync with the live platform. `openapi:gen` regenerates types; `openapi:check` fails on drift at either end (prod-vs-committed first, then committed-vs-types) — runs in CI and as ship.md gate 1; `openapi:pull` fetches the live spec and regenerates types in one step (the recovery path when `openapi:check` fails on prod drift). See the "Changing the API client" rule above for the workflow.
- **GitHub Actions CI** at `.github/workflows/ci.yml` is five jobs (`lint`, `typecheck`, `test`, `openapi`, `docker`):
  - **`lint` / `typecheck` / `test` / `openapi`** — each runs `pnpm/action-setup@v4` + `actions/setup-node@v4` (`cache: 'pnpm'`, version from `.nvmrc`) + `pnpm install --frozen-lockfile`, then the matching `pnpm check` / `pnpm typecheck` / `pnpm test:run` / `pnpm openapi:check` gate.
  - **`docker`** (`needs: quality`) — builds the multi-stage Dockerfile via `docker/build-push-action@v6` with GHA layer cache (`cache-from: type=gha` / `cache-to: type=gha,mode=max`), `push: false`. Re-runs the same gates inside the `builder` stage as a second line of defense.
  - Run-level `concurrency: ci-${workflow}-${ref}` with `cancel-in-progress: true` so only the latest commit per ref keeps running.
